# Kotlin Bytecode Patterns

## Concept

Many Android apps are written in Kotlin, and Kotlin-generated bytecode has patterns that look strange if you only learned Java-style output.

Common clues:

- `Lkotlin/jvm/internal/Intrinsics;`
- null-check helpers
- synthetic methods
- default-argument helper methods
- companion objects

Plain-English version:
Sometimes the code looks weird not because it is malicious, but because Kotlin generates extra helper logic.

Register note:
In real Smali, whether the first visible argument is `p0` or `p1` depends on whether the method is static-like or an instance method. `p0` is only `this` for non-static methods.

## Java vs. Smali

### Kotlin-style source
```kotlin
fun greet(name: String): String {
    return "Hi $name"
}
```

### Smali-style pattern
```smali
const-string v0, "name"
invoke-static {p0, v0}, Lkotlin/jvm/internal/Intrinsics;->checkNotNullParameter(Ljava/lang/Object;Ljava/lang/String;)V
new-instance v0, Ljava/lang/StringBuilder;
invoke-direct {v0}, Ljava/lang/StringBuilder;-><init>()V
```

## The Security Angle

If you mistake Kotlin compiler noise for custom app logic, you waste time.

Key analyst skill:

- separate language-generated noise from developer-written logic
